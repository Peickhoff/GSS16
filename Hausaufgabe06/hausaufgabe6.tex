\section{Zentrale Begriffe der Kryptographie}
\setcounter{subsection}{1}
\subsection{Schlüsselaustausch}
 Bei assymetrischer Verschlüsselung müssen nur für jeden Kommunikationspartner ein öffentlicher
 und ein privater Schlüssel erstellt werden. Bei n Personen also 2n Schlüssel. Für eine symmetrische
 Verschlüsselung benötigt jedes Kommunkationspaar einen eigenen Schlüssel. Bei n Personen sind dies 
 $\frac{n\cdot(n-1)}{2}$ Schlüssel.
 \subsection{Hybride Kryptosysteme}
 \begin{itemize}
 \item Wenn Alice eine sehr lange Nachricht senden möchte, dann ist die Verschlüsselung mit dem 
 relativ langen öffentlichen Schlüssel (4096-Bit) sinnvoll. Ist die Nachricht jedoch sehr lang, oder 
 will Alice sehr viele Nachrichten verschicken, ist ein hybrides Kryptosystem effizienter, da sonst sehr
 häufig mit dem langen öffentlichen Schlüssel verschlüsselt werden müsste.
 \item Um ein hybrides Kryptosystem zu benutzen, schickt Alice Bob einen mit Bobs öffentlichen Schlüssel
 verschlüsselten symmetrischen Schlüssel. Mit diesem symmetrischen Schlüsel werden dann alle 
 nachfolgenden Nachrichten verschlüsselt. Dies ist deutlich effizienter, da der symmetrische Schlüssel
 deutlicher kleiner (256-Bit) ist.

 \end{itemize}
 \section{Parkhaus}
 \setcounter{subsection}{1}
 \subsection{Sicherheitsanalyse}
 Das System ist angreifbar, da anscheinend der Barcode für den Pauschalpreis vom Kino immer identisch ist,
 und man diesen einfach auf die Karte drucken könnte. Auch der Barcode zur Bezahlbestätigung ist identisch
 , und je nachdem wie überprüft wird, ob die Karte in den letzten 10 Minuten bezahlt wurde, kann man auch
 diesen Barcode auf die Karte drucken. Außerdem können mit der selben Karte in einem 10 Minuten Fenster
 beliebig viele Autos das Parkhaus verlassen.\\\\
 \textbf{Angreifermodell:}\\
 Rolle:Außenstehender,Benutzer\\
 Verbreitung:Tickets kaufen oder benutzte Tickets auflesen\\
 Verhalten:aktiv, beobachtend (Karte kaufen oder auflesen und analysieren)\\
 Rechenkapazität:unbeschränkt\\
 \subsection{Umsetzung mit kryptographischen Techniken}
 Um die Parktickets gegen Betrug zu schützen, führen wir eine asymmetrische Verschlüsselung der Barcodes
 ein. Dafür besitzt der Ticketautomat geheime Schlüssel für alle Händler mit Sonderangeboten. Die 
 Einzelhändler nehmen den einzigartigen Barcode des Tickets (erster oder zweiter Block von rechts) und 
 verschlüsseln diesen mit ihrem öffentlichen Schlüssel. Anschlließend kann der Ticketautomat mit dem 
 zugehörigen privaten Schlüssel, den Code auf Echtheit prüfen. Diese Verschlüsselung basiert auf RSA. 
 \section{Authentifizierungsprotokolle}
 \setcounter{subsection}{1}
 \subsection{Authentifikationssytem auf Basis indeterministischer symmetrischer Verschlüsselung}
 Die indeterministische symmetrische Verschlüsselung schützt vor Angriffen auf den Klartext von Nutzername
 und Passwort. Jedoch schützt das Verfahren nicht vor einem Maskeraden-Angriff. Ein Man-In-The-Middle kann 
 das Verschlüsselte Benutzer-Passwort-Paar abfangen und dieses benutzen, um sich beim Server zu 
 authentifizieren. Da die Zufallszahl nicht vorher an den Server übertragen wird, kann der Server diese 
 auch nicht von jeder anderer Zahl unterscheiden.
 \subsection{Challenge-Response-Authentifizierug}
 Bei der Challenge-Response-Authentifizierung wird beim Verbindungsaufbau zwischen User und Server gegenseitig die Echtheit durch das Stellen und die Berechnung von sogenannten Challenges und Responses gewährleistet. Die Verbindung wird initiiert mit einer Challenge des Users, auf die der Server mit der korrekten Response und einer neuen Challenge für den User antwortet. Der User berechnet die Response für die zweite Challenge und schickt diese mit einer dritten Challenge zurück zum Server. Dieser antwortet mit einer letzten Response und ist diese korrekt, wird die Verbindung aufgebaut. Durch dieses Verfahren ist
 es nicht mehr möglich wie im vorherigen Verfahren Mittels einfacher MITM-Attacke sich als einen anderen
 Klienten auszugeben, da der Angreifer die Response nicht berechnen kann.\\
 Dieses System ist jedoch angreifbar, indem man folgendes tut:
 \begin{itemize}
 \item User schickt Challenge1 an Server1
 \item Server1 schickt Challenge2 und Response1 an User
 \item User schickt Challenge2 an Server2
 \item Server2 schickt Challenge3 und Response2 an User
 \item User schickt Response2 und Challenge3 an Server1
 \item Server1 schickt Response3 an User
 \item Verbindungsaufbau
 \end{itemize}
